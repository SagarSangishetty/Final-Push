# AWS VPC - Complete Study Notes

## 1. VPC Basics

**What is VPC?**
- VPC = Virtual Private **Cloud** (not Network)
- A virtual network within AWS where you have complete control and isolation
- Secure your resources with your own network setup

**CIDR Block**
- Defines the IP range for your VPC
- Example: `172.1.0.0/16` means 65,536 total IPs available
- `/16` = subnet mask (determines total IPs)

---

## 2. Subnets

**What are Subnets?**
- Divide your VPC into smaller networks
- Each subnet has its own CIDR block
- Example:
  - Application 1: `172.1.1.0/24`
  - Application 2: `172.1.2.0/24`

**Types of Subnets:**
- **Public Subnet** → Has route to Internet Gateway
- **Private Subnet** → No direct internet access (uses NAT Gateway)

---

## 3. Internet Gateway (IGW)

**Purpose:**
- Entry/exit point between VPC and internet
- Allows traffic to flow in both directions

**How it works:**
- Attached to VPC
- Enables public subnets to access internet
- Without it, no internet connection

---

## 4. Route Table (THE SIGN POST)

**What is it?**
- Defines the PATH for network traffic
- Tells packets where to go

**Example Route:**
- `0.0.0.0/0 → Internet Gateway`
- Meaning: Any traffic going outside → Send to IGW

**Where to use:**
- Associate route tables with subnets (public/private)
- Different subnets can have different route tables

---

## 5. Public Subnet (COMPLETE EXPLANATION)

**What makes a subnet "Public"?**
- Its route table directs traffic to Internet Gateway
- Route: `0.0.0.0/0 → IGW`

**Who can access what?**
- Internet → Can reach resources in public subnet ✅
- Resources in public subnet → Can reach internet ✅

**Three things needed for Public Subnet:**
1. **Internet Gateway** - Door to internet 🚪
2. **Route Table** - Sign post showing the path 🪧
3. **Public IP** - Address so internet finds you 📍

**Use cases:**
- Web servers (Nginx, Apache)
- Load balancers
- Bastion hosts (jump servers)
- API gateways

---

## 6. Private Subnet

**What makes a subnet "Private"?**
- Route table does NOT have direct route to IGW
- Route: `0.0.0.0/0 → NAT Gateway` (if outbound needed)

**Who can access what?**
- Internet → CANNOT reach resources ✅
- Resources → CAN reach internet (via NAT) ⚠️

**Use cases:**
- Databases (MySQL, PostgreSQL)
- Application servers
- Cache servers (Redis)
- Sensitive internal services

---

## 7. Security Group

**What is it?**
- Instance-level firewall
- Defines INBOUND and OUTBOUND rules

**Example rules:**
- Allow port 80 (HTTP) from anywhere
- Allow port 443 (HTTPS) from anywhere
- Allow port 3306 (Database) from specific IP

**Note:** Security Group is stateful (if you allow traffic in, response goes out automatically)

---

## 8. Network ACL (Additional Security Layer)

**What is it?**
- Subnet-level firewall
- Another security layer (operates at subnet level)

**Difference from Security Group:**
- Security Group = Stateful, instance-level
- NACL = Stateless, subnet-level

---

## 9. NAT Gateway (For Private Subnet Internet Access)

**What is it?**
- Allows private subnet resources to reach internet
- But internet CANNOT reach them

**Where to place:**
- In PUBLIC subnet
- Private subnet route: `0.0.0.0/0 → NAT Gateway`

**When needed:**
- Private instances need software updates
- Private instances need to call external APIs
- Private instances need outbound internet access

---

## 10. Complete VPC Setup Flow

```
Step 1: Create VPC (define CIDR: 172.1.0.0/16)
         ↓
Step 2: Create Subnets (172.1.1.0/24, 172.1.2.0/24)
         ↓
Step 3: Create Internet Gateway
         ↓
Step 4: Attach IGW to VPC
         ↓
Step 5: Create Route Tables (separate for public/private)
         ↓
Step 6: Add IGW route to public route table (0.0.0.0/0 → IGW)
         ↓
Step 7: Associate route tables with subnets
         ↓
Step 8: Create Security Groups with rules
         ↓
Step 9: Launch EC2 instances
         ↓
Step 10: Assign public IPs (to public subnet instances)
```

## 15. Quick Checklist

### For Public Subnet Setup:
- [ ] VPC created with CIDR block
- [ ] Public subnet created
- [ ] Internet Gateway attached to VPC
- [ ] Route table has: `0.0.0.0/0 → IGW`
- [ ] Route table associated with public subnet
- [ ] Security group allows inbound traffic
- [ ] EC2 instance has public IP assigned

### For Private Subnet Setup:
- [ ] VPC created with CIDR block
- [ ] Private subnet created
- [ ] NAT Gateway in public subnet (if outbound needed)
- [ ] Route table has: `0.0.0.0/0 → NAT Gateway`
- [ ] Route table associated with private subnet
- [ ] Security group defines rules
- [ ] EC2 instance (no public IP needed)

---

## 16. Common Mistakes

❌ **Mistake 1:** Thinking Route Table is optional
✅ **Fix:** Route table is ESSENTIAL - it defines traffic paths

❌ **Mistake 2:** Public IP alone makes subnet "public"
✅ **Fix:** Route table must have IGW route + Public IP + IGW attached

❌ **Mistake 3:** Using public subnet for databases
✅ **Fix:** Keep databases in private subnets for security

❌ **Mistake 4:** Not understanding why both public and private subnets are needed
✅ **Fix:** Public for user-facing, Private for backend security

---

## 17. One-Line Summaries

- **VPC** = Your virtual cloud network
- **CIDR** = IP range definition
- **Subnet** = Network divided into smaller parts
- **IGW** = Door to internet
- **Route Table** = Traffic direction sign
- **Public Subnet** = Accessible from internet (has IGW route)
- **Private Subnet** = NOT accessible from internet (no IGW route)
- **Public IP** = Internet address for your resource
- **Security Group** = Firewall with allow/deny rules
- **NAT Gateway** = Allows private subnet outbound access

---

## 18. Remember This Always

**Public Subnet = Internet ↔️ Your Resource (Bidirectional)**
**Private Subnet = Your Resource → Internet Only (One-way via NAT)**

Security has 4 layers:

NACL (Subnet-level) - Stateless firewall at subnet boundary
Security Group (Instance-level) - Stateful firewall at instance
Subnet Type (Public/Private) - Routing determines internet access
NAT Gateway - Controls outbound traffic from private subnets"**


📍 Real Example:
PRIVATE SUBNET (10.0.1.0/24)

Layer 1: NACL Rules
  Inbound Rule #100: Allow port 3306 from 10.0.0.0/24 ✅
  
Layer 2: Security Group Rules
  Inbound Rule: Allow port 3306 from sg-frontend ✅
  
Layer 3: Subnet Type
  Private subnet → No IGW attached → Hidden from internet ✅
  
Layer 4: NAT Gateway
  Route: 0.0.0.0/0 → NAT Gateway ✅
  (Allows outbound traffic only)

Result: 
- Database accessible ONLY from app servers ✅
- NOT accessible from internet ✅
- Can reach internet for updates (via NAT) ✅
