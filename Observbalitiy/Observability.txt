Observability
Observability is the ability to understand the internal state of a system from its external outputs — without needing to physically inspect it.
The goal is to answer:
What is happening?   → Metrics
Why is it happening? → Logs
How did it happen?   → Traces

3 Pillars
Metrics (What)
Numerical measurements collected over time. Tells you the current and historical state of your system.
Examples:
- CPU utilization of a node in last 30 min
- Memory usage of a pod
- Disk size of a node
- Number of HTTP requests per second
- Pod restart count
Key point — Metrics tell you something is wrong but not why.

Logging (Why)
Text records of events generated by your application or system. Helps you understand why something happened.
Examples:
- Why did memory utilization increase?
  → Log shows: "Loading 500MB config file on every request"
- Why did CPU spike at 10:00?
  → Log shows: "Batch job triggered at 10:00"
- Why did the pod restart?
  → Log shows: "OOMKilled — out of memory"
Key point — Logs tell you the reason behind the metric change.

Traces (How)
Tracks the journey of a single request across multiple services. Tells you exactly where and how a request failed.
Example:
HTTP request failed at 10:00 AM

Trace shows the full path:
Browser → nginx (2ms) → backend API (850ms) → RDS (timeout after 30s)

Identified: RDS connection pool was exhausted
Root cause found in 2 minutes instead of 2 hours
Key point — Traces answer how did this specific request fail and where.

How All 3 Work Together
Metrics alert you  → "Error rate spiked at 10:00 AM"
       ↓
Logs explain why   → "DB connection refused — pool exhausted"
       ↓
Traces show where  → "Request hung at backend → RDS step for 30s"
       ↓
Root cause identified → "RDS max_connections limit hit"
Without all 3 you have an incomplete picture. This is why observability needs all 3 pillars together.

Real Interview Answer
If asked "What is Observability?":

"Observability is the ability to understand the internal state of a system through its external outputs. It has 3 pillars — Metrics tell you what is happening, Logs tell you why it is happening, and Traces tell you how a specific request failed across services. In my project I implemented CloudWatch Container Insights for metrics, Fluent Bit for log shipping, and health check endpoints to verify system state."
